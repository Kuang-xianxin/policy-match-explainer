# 龙华区企业库可行性方案

日期：2026-07-03

## 结论

可以做一个服务于本项目的“龙华企业库”，但不能把它理解成免费、无限制、实时复制一份外部工商全量库。

更准确的产品定义是：

- 建立本项目自己的 `longhua_enterprise_registry` 索引库。
- 数据来自合法授权的数据源，例如深圳政府数据开放平台、商业企业数据 API、用户授权上传或后续人工维护数据。
- 本地数据库只保存政策匹配需要的字段、来源、同步时间、原始快照和字段可信度。
- DeepSeek 只能做查询规划、字段解耦、行业/主营业务归一化和缺失字段识别，不能作为企业事实来源。

## 不建议做的方案

不建议直接爬取国家企业信用信息公示系统、深圳市市场监管公示平台、深圳信用网等查询页面来构建全量库。

原因：

- 这些页面通常面向查询和公示，不等同于允许无约束批量采集。
- 可能存在验证码、登录、限频、反自动化和使用条款限制。
- 绕过访问限制会带来合规风险，也会导致系统不稳定。
- 财务、纳税、从业人数等字段在法规层面存在“企业选择是否公示”的情况，不能假定全量可得。

## 推荐数据来源

### 1. 深圳政府数据开放平台

优先验证深圳政府数据开放平台中与统一社会信用代码、证照、市场主体相关的数据接口。

可用价值：

- 用于企业名称、统一社会信用代码、登记/证照基础信息校验。
- 适合做官方来源的基础企业索引。
- 平台数据接口通常可按 API 方式接入，工程稳定性比爬网页高。

限制：

- 需要逐个确认接口字段、开放级别、调用限制、是否支持按区筛选和批量同步。
- 政府开放数据不一定覆盖企业画像所需的所有字段。
- 仍需遵守平台服务条款，标注数据来源，不得歪曲篡改数据。

参考：

- 深圳政府数据开放平台统一社会信用代码信息：https://opendata.sz.gov.cn/data/api/toApiDetails/29200_01300275
- 深圳政府数据开放平台证照类接口示例：https://opendata.sz.gov.cn/data/api/toApiDetails/29200_01303625

### 2. 商业企业数据 API

如果目标是“尽量接近龙华全量企业库”，商业 API 是更现实的方案，例如企查查开放平台。

可用价值：

- 企业搜索、工商信息、风险、知识产权、经营状况等字段覆盖更完整。
- 通常适合做企业名称自动补全、企业候选列表、基础画像生成。
- 如果供应商支持离线数据或批量接口，可以做定期全量/增量同步。

限制：

- 通常需要采购，字段授权、调用量、缓存期限、二次使用范围需要合同确认。
- 第三方数据仍要保存来源和更新时间，不能替代用户对申报材料真实性的确认。

参考：

- 企查查开放平台：https://openapi.qcc.com/
- 企查查工商接口 API：https://mapi.qcc.com/dataApi

### 3. 官方公示/信用查询入口

国家企业信用信息公示系统、深圳市市场监管公示平台、深圳信用网适合作为单企核验、人工复核或低频查询入口，不适合作为第一版批量建库来源。

可用价值：

- 用于核验企业年报、抽查检查、行政处罚、经营异常等信用信息。
- 可在报告中展示“需要人工复核”的来源链接。

限制：

- 查询入口可能需要验证码、登录或交互验证。
- 不建议绕过限制批量抓取。

参考：

- 深圳市市场监督管理局商事主体信用监管公示平台：https://amr.sz.gov.cn/xyjggs.webui/xyjggs/index.aspx
- 深圳信用网：https://www.szcredit.org.cn/
- 企业信息公示暂行条例：https://www.stats.gov.cn/gk/tjfg/xgfxfg/202503/t20250306_1958886.html

## 推荐技术方案

### 数据库分层

新增一个项目级企业索引库，不与用户手动画像混在一起。

建议新增表：

```sql
longhua_enterprise_registry
```

核心字段：

- `id`
- `company_name`
- `credit_code`
- `registration_status`
- `registered_address`
- `district`
- `street`
- `registered_capital`
- `established_date`
- `business_scope`
- `industry_raw`
- `industry_normalized`
- `source_name`
- `source_type`
- `source_url`
- `raw_payload`
- `profile_draft`
- `field_sources`
- `data_quality_score`
- `last_synced_at`
- `created_at`
- `updated_at`

建议新增同步任务表：

```sql
enterprise_registry_sync_jobs
```

核心字段：

- `id`
- `provider_name`
- `sync_mode`
- `started_at`
- `finished_at`
- `status`
- `fetched_count`
- `inserted_count`
- `updated_count`
- `failed_count`
- `error_message`

### 查询流程

1. 用户输入企业名称。
2. DeepSeek 生成查询计划和关键词。
3. 后端先查 `longhua_enterprise_registry` 本地索引。
4. 本地命中时直接返回候选企业。
5. 本地未命中时，根据配置调用外部 provider。
6. 外部 provider 命中后写入本地索引，并保存 `raw_payload`。
7. DeepSeek 基于 `raw_payload` 解耦为企业画像草稿。
8. 用户确认后写入 `enterprise_profiles`。

### 同步策略

第一阶段不要一开始追求“全量龙华所有企业”，先做可验证的小闭环：

1. 手工导入 100-500 条测试企业，验证字段、搜索、解耦、匹配流程。
2. 接入一个官方开放数据接口，做按页同步或按关键词/时间同步。
3. 如果官方接口不足，再采购商业 API 做企业搜索和详情补全。
4. 如果供应商支持离线库，再做龙华区全量初始导入。
5. 每日或每周增量刷新经营状态、地址、资质、风险字段。

## 龙华区过滤规则

不能只用企业名称判断是否属于龙华区。

推荐过滤条件：

- `registered_address` 包含“深圳市龙华区”。
- 或数据源提供的行政区字段为“龙华区”。
- 或数据源提供登记机关/辖区字段可映射到龙华区。
- 如果地址只写深圳市但无区级信息，标记为 `district_confidence = low`，需要用户确认。

## 企业画像字段覆盖边界

企业库可以比较可靠地覆盖：

- 企业名称
- 统一社会信用代码
- 登记/注册地址
- 成立日期
- 注册资本
- 经营状态
- 经营范围
- 粗略行业
- 部分资质名单
- 部分风险/处罚/异常信息

企业库通常不能可靠覆盖：

- 上年营收
- 上年利润
- 上年纳税
- 研发费用
- 研发人员数
- 社保是否正常
- 真实客户类型
- 申报项目名称、阶段、预算

这些字段仍应由用户确认或上传材料补充。DeepSeek 不得猜测。

## 建议落地顺序

### M1：本地企业索引表

- 新增 `longhua_enterprise_registry`。
- 把当前 demo 企业迁入这个表。
- `/api/company-lookup/search` 优先查询本地索引。
- 前端继续展示候选来源和置信度。

### M2：CSV/Excel 导入

- 做一个仅开发/管理员使用的导入脚本。
- 支持导入采购数据、开放数据下载文件或人工整理名单。
- 导入时生成字段来源和同步记录。

### M3：官方开放数据 provider

- 接入深圳政府数据开放平台中已确认可用的数据接口。
- 做分页、限流、失败重试、同步日志。
- 只保存政策匹配需要的字段。

### M4：商业 API provider

- 在拿到 API Key 和授权条款后接入企查查等商业接口。
- 支持企业名称搜索、企业详情补全、风险/知识产权字段补充。
- 按合同要求控制缓存时间、展示范围和调用频率。

### M5：全量/增量维护

- 如果拿到离线库或批量接口，做龙华区全量初始导入。
- 后续按日/周增量刷新。
- 对注销、迁出、异常经营等状态做历史保留，不直接覆盖删除。

## 对当前项目的建议

可以做，但第一版不要承诺“拥有龙华所有企业所有画像字段”。

建议产品文案改成：

> 系统维护龙华区企业基础信息索引库，用于企业名称搜索、公开字段补全和政策匹配预填。企业经营、财务、研发、社保、申报项目等字段仍需用户确认或补充。

这样既能提升体验，又不会把公开/商业数据能力夸大成完整画像自动生成。
